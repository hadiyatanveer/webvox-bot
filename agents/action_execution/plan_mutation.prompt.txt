====================================================
CORE MUTATION CONCEPTS
====================================================

1. **Handle Not-NULL Constraints (CRITICAL)**:
   - If a column is REQUIRED but missing from the input, you MUST provide a value.
   - **Timestamps**: Use `{current_time}` for columns like `created_at` or `order_date`.
   - **Calculated Fields**: Use a reasonable default (e.g., `total: 0.0`) or derive it if possible (e.g., `total: price * quantity`).
   - **Status**: Use "pending" or another standard default for the operation.

2. **Entity & Foreign Key Resolution (CRITICAL)**:
   - **Linking via Unique Attribute**: If you have a unique attribute (like name: "Margherita Pizza") but NOT the ID (menu_item_id), use a nested insert with `on_conflict` that does nothing but return the ID.
   - **Nested Lookup Pattern**:
     ```graphql
     relationship_name: {{
       data: {{ unique_column: "value" }},
       on_conflict: {{ constraint: table_unique_key, update_columns: [] }}
     }}
     ```
   - **DO NOT GUESS IDs**. Use this pattern when you have a name/identifier but no numeric ID.

3. **Insert One (`insert_TABLE_one`)**:
   - Used for primary table operations. Accepts an `object` argument.

4. **Nested Inserts (ARRAY Relationships)**:
   - To insert multiple related rows, use the relationship name with a `data` array.
   - Example: `order_items: {{ data: [ {{ quantity: 1, ... }} ] }}`

5. **Update (`update_TABLE`)**:
   - Requires a `where` clause and a `_set` clause.
   - ALWAYS include security filters (user_id) in the `where` clause.

====================================================
GUIDELINES
====================================================

1. **User ID Enforcement**:
   - **Insert**: ALWAYS set `user_id: {user_id}` if the table has it.
   - **Update**: ALWAYS add `user_id: {{ _eq: {user_id} }}` to the `where` clause.

2. **Auto-IDs**: NEVER attempt to set or update auto-incrementing Primary Key columns (like `id`).
