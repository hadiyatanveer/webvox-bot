====================================================
CORE CONCEPTS — READ CAREFULLY
====================================================

1. OBJECT relationships (many-to-one):
   - Returns a SINGLE related row.
   - Does NOT accept a 'where' argument at selection level.
   - To FILTER by an OBJECT relationship, use it inside the PARENT table's where clause:
       menu_items(where: {{ categories: {{ name: {{ _ilike: "%Pizza%" }} }} }}) {{ ... }}
   - To SELECT fields, just nest the name without a where:
       menu_items {{ categories {{ name }} }}

2. ARRAY relationships (one-to-many):
   - Returns MULTIPLE related rows.
   - DOES accept a 'where' argument at selection level:
       categories {{ menu_items(where: {{ is_vegetarian: {{ _eq: true }} }}) {{ name }} }}

====================================================
MULTI-HOP / NESTED RELATIONSHIPS — CRITICAL
====================================================

You MUST support arbitrarily deep relationship chains. A request can span 3, 4, or more levels.

EXAMPLE CHAIN: users → orders → order_items → menu_items → categories

Each hop is a relationship defined on the CURRENT table in the chain.
- 'orders' is a relationship on 'users'
- 'order_items' is a relationship on 'orders'
- 'menu_items' is a relationship on 'order_items'
- 'categories' is a relationship on 'menu_items'

To express this in the JSON plan, use RECURSIVE nesting inside the "relationships" array.
Each relationship object CAN contain its own "relationships" array for the next hop.

NESTED RELATIONSHIP FORMAT:
{{
  "name": "orders",
  "fields": ["id", "total", "status"],
  "where": {{}},
  "relationships": [
    {{
      "name": "order_items",
      "fields": ["id", "quantity", "price_at_order"],
      "where": {{}},
      "relationships": [
        {{
          "name": "menu_items",
          "fields": ["id", "name", "price"],
          "where": {{}},
          "relationships": [
            {{
              "name": "categories",
              "fields": ["id", "name"],
              "where": {{}},
              "relationships": []
            }}
          ]
        }}
      ]
    }}
  ]
}}

RULES FOR NESTING:
- NEVER skip a hop. If you need users → orders → order_items, you MUST include all three levels.
- Each "name" must be a relationship that exists on the table at THAT level, not the root table.
- "id" should always be included in "fields" at each level.
- An empty "relationships": [] signals the end of a chain.

====================================================
WHERE CLAUSE OPERATORS
====================================================

- _eq        → Exact match
- _neq       → Not equal
- _ilike     → Case-insensitive match (use % for wildcards)
- _in        → Value in array
- _gt, _gte  → Greater than
- _lt, _lte  → Less than
- _is_null   → Check for null
- _and, _or, _not → Logical operators

Rules:
- Boolean fields: true/false (not strings)
- Int fields: whole numbers only
- numeric fields: floats allowed (e.g., 9.99)
- Use _ilike for partial text matches
- Use _eq for exact matches
