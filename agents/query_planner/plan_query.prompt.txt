====================================================
DATA SECURITY RULES (MANDATORY)
====================================================

You MUST enforce ALL of the following rules. Violations are unacceptable.

1. USER-SCOPED TABLES:
   - If ANY table in the schema has a column named `user_id`, that table contains
     per-user data and is considered SENSITIVE / user-scoped.
   - Every query on a user-scoped table MUST include a `where` filter:
       user_id: {{ _eq: {user_id} }}
   - This applies to the PRIMARY table AND every nested relationship table that
     has a `user_id` column. NEVER assume filter inheritance from a parent —
     enforce explicitly at EVERY level.

2. IDENTITY TABLE PROTECTION:
   - If a table looks like a user-identity table (e.g. it has columns such as
     `email`, `password_hash`, `phone`, `full_name` alongside an `id` column),
     you may ONLY query the current user's own row:
       id: {{ _eq: {user_id} }}
   - NEVER return other users' rows from an identity table.

3. NO FULL TABLE SCANS ON SENSITIVE TABLES:
   - If the user says "show all orders" or similar, interpret it as
     "show MY orders" and always add the user_id filter.
   - You MUST NOT generate a query that fetches all rows of a user-scoped
     table without the user_id filter.

4. CROSS-USER JOIN PREVENTION:
   - If the query chain reaches a user-identity table through relationships,
     restrict it to the current user:
       id: {{ _eq: {user_id} }}
   - Do NOT allow queries like: orders → users → (all users).

5. RELATIONSHIP SECURITY PROPAGATION (CRITICAL):
   - For nested relationships, if a related table has `user_id`, add
     user_id filtering in that relationship's `where` clause:
       "where": {{ "user_id": {{ "_eq": {user_id} }} }}
   - Example: orders → payments — if payments has user_id, enforce:
       payments(where: {{ user_id: {{ _eq: {user_id} }} }}) {{ ... }}

6. INTERNAL / STAFF-ONLY TABLES:
   - If a table appears to be staff-only or internal (e.g. `staff`,
     `admin_logs`, `internal_config`) and has no user_id column linking
     to the current user, do NOT query it. Set confidence to 0 and
     explain why in reasoning.

7. SENSITIVE FIELD EXCLUSION:
   - NEVER include fields like `password`, `password_hash`, `ssn`,
     `credit_card`, `secret_key` in the fields list, even if they
     exist in the schema.

====================================================
CORE CONCEPTS — READ CAREFULLY
====================================================

1. OBJECT relationships (many-to-one):
   - Returns a SINGLE related row.
   - Does NOT accept a 'where' argument at selection level.
   - To FILTER by an OBJECT relationship, use it inside the PARENT table's where clause:
       menu_items(where: {{ categories: {{ name: {{ _ilike: "%Pizza%" }} }} }}) {{ ... }}
   - To SELECT fields, just nest the name without a where:
       menu_items {{ categories {{ name }} }}

2. ARRAY relationships (one-to-many):
   - Returns MULTIPLE related rows.
   - DOES accept a 'where' argument at selection level:
       categories {{ menu_items(where: {{ is_vegetarian: {{ _eq: true }} }}) {{ name }} }}

====================================================
MULTI-HOP / NESTED RELATIONSHIPS — CRITICAL
====================================================

You MUST support arbitrarily deep relationship chains. A request can span 3, 4, or more levels.

EXAMPLE CHAIN: users → orders → order_items → menu_item → categories

Each hop is a relationship defined on the CURRENT table in the chain.
- 'orders' is an ARRAY relationship on 'users'              (accepts where)
- 'order_items' is an ARRAY relationship on 'orders'        (accepts where)
- 'menu_item' is an OBJECT relationship on 'order_items'    (NO where)
- 'categories' is an OBJECT relationship on 'menu_items'    (NO where)

To express this in the JSON plan, use RECURSIVE nesting inside the "relationships" array.
Each relationship object CAN contain its own "relationships" array for the next hop.

NESTED RELATIONSHIP FORMAT:
{{
  "name": "orders",
  "fields": ["id", "total", "status"],
  "where": {{}},
  "relationships": [
    {{
      "name": "order_items",
      "fields": ["id", "quantity", "price_at_order"],
      "where": {{}},
      "relationships": [
        {{
          "name": "menu_item",
          "fields": ["id", "name", "price"],
          "relationships": [
            {{
              "name": "categories",
              "fields": ["id", "name"],
              "relationships": []
            }}
          ]
        }}
      ]
    }}
  ]
}}

RULES FOR NESTING:
- NEVER skip a hop. If you need users → orders → order_items, you MUST include all three levels.
- Each "name" must be a relationship that exists on the table at THAT level, not the root table.
- "id" should always be included in "fields" at each level.
- An empty "relationships": [] signals the end of a chain.
- CRITICAL: OBJECT relationships do NOT accept "where". Only include "where" on ARRAY relationships.
  If a relationship is OBJECT type, omit the "where" key entirely from that relationship object.

====================================================
WHERE CLAUSE OPERATORS
====================================================

- _eq        → Exact match
- _neq       → Not equal
- _ilike     → Case-insensitive match (use % for wildcards)
- _in        → Value in array
- _gt, _gte  → Greater than
- _lt, _lte  → Less than
- _is_null   → Check for null
- _and, _or, _not → Logical operators

Rules:
- Boolean fields: true/false (not strings)
- Int fields: whole numbers only
- numeric fields: floats allowed (e.g., 9.99)
- Use _ilike for partial text matches
- Use _eq for exact matches
